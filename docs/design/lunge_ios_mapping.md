# lunge (弓箭步) 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/lunge_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 首要考虑复用 iOS 已有的检测项，若无匹配项再分配新 `itemID`。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对静态动作的特殊性，提供明确的状态机实例化与多条件 AND 计时方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- 静态动作不采用动态动作的 `P1/P2` 状态机，而是采用基于 `Holding/Resting` 的多条件 AND 计时状态机。
- 严格遵守 `[min, max, needs_absolute, debounce]` 4 槽位协议。

## 动作业务描述与覆盖结论

**弓箭步 (lunge)** 是一项**典型的矢状面单侧发力静态动作**。
在我们的算法设计中，弓箭步的评估核心在于“前侧支撑腿”的下蹲深度与发力状态。由于算法内部（`MetricsCalculator`）能够通过动态判断或配置，自动找到在画面中处于“前方”或者“主要发力”的那条腿，因此**在 iOS 端无需拆分左腿和右腿检测项**。一套通用的检测参数即可兼容“左腿在前”和“右腿在前”以及“人物朝左”和“人物朝右”的所有情况。

### 训练产物核心检测项（`metrics`）
1. `knee_flexion` (膝关节屈曲) - **核心下蹲深度指标**（评价前侧支撑腿膝盖是否弯曲到位，目标约 95°）。
2. `trunk_lean` (躯干倾斜角) - **核心姿态指标**（评价身体是否直立，防止过度前倾压迫膝盖）。
3. `hip_flexion` (髋关节屈曲) - **核心姿态指标**（与膝关节配合，评价髋部的下沉幅度，目标约 94°）。

### iOS 复用/新增判定与影响分析

- `trunk_lean` -> **强烈建议复用 `itemID=39` (Trunk Lean Static Hold)**。
  - *原因分析*：我们在平板支撑和哥本哈根支撑中都已经使用了 `itemID=39` 来评估躯干倾斜度。虽然弓箭步要求身体直立（接近 0° 倾斜，区别于平板支撑的接近 90°），但底层的算法逻辑（肩中心到髋中心与重力线的夹角）是完全一致的，仅仅是传入的 `[min, max]` 参数不同。**完全可复用**。
- `hip_flexion` -> **强烈建议复用 `itemID=40` (Hip Flexion Static Hold)**。
  - *原因分析*：同样在前面的动作中定义了用于评估髋关节夹角的静态策略。底层逻辑（肩-髋-膝的夹角）通用，且由于算法自动选取主发力腿，只需下发对应的弓箭步参数即可。**完全可复用**。
- `knee_flexion` -> **建议新增 `itemID=46` (Knee Flexion Static Hold)**。
  - *原因分析*：这是目前静态动作库中第一次引入专门针对膝关节角度的纯静态评估（动态动作中有深蹲的动态膝盖评估，但不能混用）。由于需要适配 4 槽位的 `[min, max, needs_absolute, debounce]` 静态计时逻辑，必须为其分配一个专用的静态策略 ID。

## 逐字段 -> 逐索引映射

### 静态 4 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平左右/朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（统一填 15.0 帧，即 0.5 秒，表示需连续达标 0.5s 才进入 Holding） |

---

### itemID=46（Knee Flexion，膝关节屈曲/下蹲深度）- 新增
- 来源字段：
  - `normal_range` = `[75.42, 127.37]`
  - `needs_absolute` = `0.0` (夹角天然无向，无需绝对值)

| iOS item46 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `75.42` | normal_range 下界 (蹲得最深的极限) |
| `[1]` | `127.37` | normal_range 上界 (限制站得太直) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[75.42, 127.37, 0.0, 15.0]`

---

### itemID=39（Trunk Lean，躯干倾斜角）- 复用
- 来源字段：
  - `normal_range` = `[-3.59, 18.42]` (转换为绝对值后建议设定为 `[0.0, 18.5]`)
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容朝向和前/后倾的符号差异)

由于开启了绝对值处理，我们取绝对值的最大范围作为上限。

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `18.5` | normal_range 上界 (允许躯干最多前倾或后仰 18.5度) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[0.0, 18.5, 1.0, 15.0]`

---

### itemID=40（Hip Flexion，髋关节屈曲）- 复用
- 来源字段：
  - `normal_range` = `[73.55, 125.58]`
  - `needs_absolute` = `0.0` (夹角天然无向)

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `73.55` | normal_range 下界 |
| `[1]` | `125.58` | normal_range 上界 |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[73.55, 125.58, 0.0, 15.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（下蹲深度、躯干直立、髋部下沉）
```json
{
  "detectItemIDs": ["46", "39", "40"],
  "detectItemParameters": [
    [75.42, 127.37, 0.0, 15.0],
    [0.0, 18.5, 1.0, 15.0],
    [73.55, 125.58, 0.0, 15.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：`itemID=39` (Trunk Lean), `itemID=40` (Hip Flexion)。只需下发新的参数。
- **需新增开发项**：
  - `itemID=46`（Knee Flexion 静态检测项：用于纯静态逻辑的膝关节夹角计算，继承动态动作中计算 `hip-knee-ankle` 夹角的方法，但走静态 4 槽位计时逻辑）。
  - **Action 总控层**：确保这 3 个检测项执行强 AND 逻辑。

## 验收点与防作弊/熔断逻辑
1. **下蹲深度熔断 (Knee & Hip)**：用户站立时（膝关节角度接近 180°），绝对不能开始计时。只有当身体下沉，前侧膝盖和髋关节角度双双弯曲到 127° 和 125° 以下时，才能触发准备状态。若中途站起，立即熔断暂停。
2. **身体过度前倾熔断 (Trunk Lean)**：部分用户在做弓箭步时由于腿部力量不足，会将上身狠狠压在前腿上。此时 `trunk_lean` 绝对值会激增。一旦躯干偏离垂直线超过 18.5 度，必须立刻熔断暂停，提示“保持上身直立”。
3. **无视朝向与前后腿验证**：测试人员应分别尝试“左腿在前”和“右腿在前”，并分别尝试面朝屏幕左侧和右侧。系统必须在不需要任何手动切换的前提下，通过 `needs_absolute` 和自动侧面选择逻辑稳定计入时长。
4. **抗抖动（Debounce）验证**：下蹲过程和站起过程中，计时器不能乱跳。只有完全沉降到位且保持稳定 0.5 秒（15帧）后，计时器才开始累加。
