# single_leg_stance 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/single_leg_stance_static_config.json` 与本地验证脚本，给出可直接执行的 iOS 迁移方案。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对静态动作（Static Action）的特殊性，提供明确的状态机实例化与多条件 AND 计时方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- 静态动作不采用动态动作的 `P1/P2` 状态机，而是采用基于 `Holding/Resting` 的多条件 AND 计时状态机。
- 严格遵守 `[min, max, needs_absolute, debounce]` 4 槽位协议。

## 动作业务描述与覆盖结论

**单脚站立 (single_leg_stance)** 是一项**双侧对称的全局静态动作**。虽然名字叫“单脚”，但在算法层面上，我们通过自动选取活动侧（抬起的脚）和取绝对值的方式，实现了**动作的全局统一评估，无需区分左右脚**。即：无论用户抬起左脚还是右脚，都使用同一套检测项配置。

### 训练产物核心检测项（`metrics`）
1. `leg_elevation_height` (腿部抬升高度) - **核心离地指标**（确保一只脚明显抬离地面）。
2. `pelvic_obliquity` (骨盆倾斜角) - **核心姿态指标**（评价单脚支撑时骨盆是否过度塌陷代偿）。
3. `trunk_lateral_flexion` (躯干侧倾角) - **核心姿态指标**（评价单脚支撑时躯干是否过度侧弯代偿）。

### iOS 复用/新增判定与影响分析

由于算法层的自适应特性，这三个检测项都**不需要**拆分为左/右两个 ID。我们为它们分配全局唯一的静态检测 ID：

- `leg_elevation_height` -> **建议分配 `itemID=42` (Leg Elevation Static Hold)**。
  - *原因分析*：计算逻辑会自动寻找被抬起的一侧（Y轴变化较大的一侧）作为抬起腿，并与支撑腿比较高度比例。无论抬左脚还是右脚，都复用这个逻辑，因此只需一个 ID。
- `pelvic_obliquity` -> **建议分配 `itemID=43` (Pelvic Obliquity Static Hold)**。
  - *原因分析*：正面骨盆倾斜角度。通过配置 `needs_absolute=1.0` 抹平向左塌陷或向右塌陷的符号差异，只需一个 ID。
- `trunk_lateral_flexion` -> **建议分配 `itemID=44` (Trunk Lateral Flexion Static Hold)**。
  - *原因分析*：正面躯干侧弯角度。同样通过配置 `needs_absolute=1.0` 抹平向左侧弯或向右侧弯的符号差异，只需一个 ID。

## 逐字段 -> 逐索引映射

### 静态 4 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平左右/朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（统一填 15.0 帧，即 0.5 秒，表示需连续达标 0.5s 才进入 Holding） |

---

### itemID=42（Leg Elevation Height，腿部抬升高度）
- 来源字段：
  - `normal_range` = `[0.10, 0.85]` (数值越小抬得越高，1.0为落地)
  - `needs_absolute` = `0.0` (高度差比例天然为正，无需绝对值)

| iOS item42 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.10` | normal_range 下界 (避免异常极小值) |
| `[1]` | `0.85` | normal_range 上界 (必须严格小于1.0以确保脚离地) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 (按30fps计，填 15.0 帧，或根据 iOS 实际定义填秒数 0.5) |

**建议参数行：**
- `[0.10, 0.85, 0.0, 15.0]`

---

### itemID=43（Pelvic Obliquity，骨盆倾斜角）
- 来源字段：
  - `normal_range` = `[0.0, 25.0]`
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容左右侧骨盆塌陷)

| iOS item43 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `25.0` | normal_range 上界 (倾斜容忍度25度) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[0.0, 25.0, 1.0, 15.0]`

---

### itemID=44（Trunk Lateral Flexion，躯干侧倾角）
- 来源字段：
  - `normal_range` = `[0.0, 25.0]`
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容左右侧躯干代偿)

| iOS item44 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `25.0` | normal_range 上界 (侧弯容忍度25度) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[0.0, 25.0, 1.0, 15.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（含抬脚判定、骨盆约束与躯干约束）
```json
{
  "detectItemIDs": ["42", "43", "44"],
  "detectItemParameters": [
    [0.10, 0.85, 0.0, 15.0],
    [0.0, 25.0, 1.0, 15.0],
    [0.0, 25.0, 1.0, 15.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无
- **需新增开发项**：
  - `itemID=42`（Leg Elevation 静态检测项：需内置自动判断抬起侧逻辑，即自动选取 Y 轴坐标更高的一侧脚踝进行比例计算）。
  - `itemID=43`（Pelvic Obliquity 静态检测项：计算左右髋连线与水平面的夹角，需读取 `[2]` 槽位决定是否取绝对值）。
  - `itemID=44`（Trunk Lateral Flexion 静态检测项：计算肩中心与髋中心连线与垂直面的夹角，需读取 `[2]` 槽位决定是否取绝对值）。
  - **Action 总控层**：确保这 3 个检测项执行强 AND 逻辑，同时达标才走动计时器。

## 验收点与防作弊/熔断逻辑
1. **双脚站立原地熔断**：用户双脚踩在地面上时，`itemID=42` 计算出的比例会接近 1.0，大于配置的 `0.85` 上界，因此计时器绝对不能启动。必须要求脚明显抬离地面。
2. **极度扭曲/代偿熔断**：当用户虽然抬起了一只脚，但为了保持平衡导致身体极度歪斜（躯干侧弯超过 25 度）或胯部极度塌陷（骨盆倾斜超过 25 度）时，`itemID=43` 或 `itemID=44` 会触发拦截，总计时器必须立即暂停，直到姿势恢复端正。
3. **无视抬脚侧验证**：要求测试人员分别抬起左脚和右脚进行测试。由于 `itemID=42` 的自适应逻辑和 `43/44` 的绝对值逻辑，无论抬哪只脚，都必须能稳定计入时长，不需要手动切换动作模式。
4. **抗抖动（Debounce）验证**：抬脚瞬间由于身体晃动，可能会有短暂的不达标。必须确保在进入标准姿态后，稳定保持 0.5 秒（即配置的 15 帧）以上，计时器才开始累加；姿态一旦崩溃，需立刻停止计时。
