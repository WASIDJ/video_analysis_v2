# hamstring_stretch (腘绳肌拉伸) 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/hamstring_stretch_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 延续我们在“股四头肌拉伸”中引入的 `SideSelectionMode` 5 槽位策略协议，解决纯静态拉伸动作中主导侧（被拉伸腿）的自动寻边与锁定问题。
- 给出可用于后端人工配置的 JSON 示例。

## 关键边界与协议说明

### 5 槽位协议 (`[min, max, needs_absolute, debounce, side_selection_mode]`)
同股四头肌拉伸一样，腘绳肌拉伸也是一个单侧静态动作（支撑腿微曲，拉伸腿伸直），无法依赖动态的 Y 轴运动幅度来选腿。
我们将继续使用第 5 槽位 `side_selection_mode`（选边模式）：
- `0.0`: 默认模式（动态找腿）。
- `1.0`: 静态启发式 - 选角度最小（最弯曲）的一侧。
- `2.0`: **静态启发式 - 选角度最大（最伸直）的一侧 (Max Angle)。** 本动作的核心！

## 动作业务描述与覆盖结论

**腘绳肌拉伸 (hamstring_stretch)** 是一项**矢状面单侧发力的静态拉伸动作**。
它要求单腿在前方尽量伸直（脚跟着地脚尖勾起），另一条腿在后方微微弯曲作为支撑，同时躯干保持平直地向下前倾。
由于这又是一个极度不对称的静态动作，我们在 iOS 端无需拆分左腿和右腿检测项。一套通用的检测参数，结合 `side_selection_mode = 2.0`（找最直的那条腿），即可兼容左右腿及朝向的所有情况。

### 训练产物核心检测项（`metrics`）
1. `knee_flexion` (膝关节屈曲) - **核心指标**（评价拉伸腿是否充分伸直，不伸直拉伸感会大减）。
2. `trunk_lean` (躯干倾斜角) - **核心指标**（评价上身是否压下去了）。
3. `hip_flexion` (髋关节屈曲) - **辅助指标**（评价是否以髋为轴折叠）。

### iOS 复用判定与分析（100% 复用！）

在这套设计下，所有的指标都完美复用了我们在前几个动作中规划的静态检测项。

- `trunk_lean` -> **复用 `itemID=39` (Trunk Lean Static Hold)**。
  - 全局指标算脊柱，不涉及选腿，第 5 槽位填 `0.0`。
- `knee_flexion` -> **复用 `itemID=46` (Knee Flexion Static Hold)**。
  - 双侧指标，由于我们要追踪的是“被拉伸的（尽量伸直的）”那条腿，因此第 5 槽位必须下发 **`2.0`**（找角度最大的一侧）。
- `hip_flexion` -> **复用 `itemID=40` (Hip Flexion Static Hold)**。
  - 双侧指标，跟随 `knee_flexion` 找到的那条伸直腿计算髋部折叠。第 5 槽位同样下发 **`2.0`**。

## 逐字段 -> 逐索引映射

### itemID=46（Knee Flexion，膝关节屈曲）- 复用
- 来源字段：
  - `normal_range` = `[136.53, 178.96]` (拉伸腿必须伸直，但允许稍微弯一点到 136 度，上限可放宽至 185 容忍检测过伸)
  - `needs_absolute` = `0.0`

| iOS item46 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `136.53` | normal_range 下界 |
| `[1]` | `185.0` | normal_range 上界 (允许微小的过伸检测误差) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `2.0` | **核心：指示策略强制去寻找左/右膝中【角度更大（更直）】的那条腿作为拉伸腿** |

**建议参数行：**
- `[136.53, 185.0, 0.0, 15.0, 2.0]`

- 来源字段：

  - `needs_absolute` = `1.0`

- 来源字段：

  - `needs_absolute` = `0.0`

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `76.87` | normal_range 下界 |
| `[1]` | `140.07` | normal_range 上界 |

| `[3]` | `15.0` | 0.5秒防抖 |

## 外部配置示例（可人工配置）

### 完整覆盖（含前腿伸直、躯干下压、髋部折叠）
```json
{
  "detectItemIDs": ["46", "39", "40"],
  "detectItemParameters": [
    [136.53, 185.0, 0.0, 15.0, 2.0],
    [12.32, 52.08, 1.0, 15.0, 0.0],
    [76.87, 140.07, 0.0, 15.0, 2.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无。
- **需新增开发项（底层协议升级补充）**：
  - 在股四头肌拉伸中，我们要求 iOS 端开发 `mode == 1.0`（找最小角度）的逻辑。
  - 在这个动作中，我们要求 iOS 端将该逻辑补全，支持 **`mode == 2.0`**（找最大角度）。
  - **状态机寻边触发条件**：同理，处于寻边期时，判断 `abs(leftAngle - rightAngle) > 20.0`。一旦大于 20 度，立刻将 `lockedSide` 赋值为角度**较大**的那一条腿（`leftAngle > rightAngle ? .left : .right`），并死锁追踪。

## 验收点与防作弊/熔断逻辑
1. **拉伸腿识别验收（核心）**：测试人员故意把一条腿伸得很直（170度），另一条腿弯曲（120度）。因为下发了 `side_selection_mode = 2.0`，系统必须立刻锁定那条伸直的腿。
2. **屈膝代偿熔断 (Knee Flexion)**：用户拉伸时如果觉得太疼，把拉伸腿弯曲（小于 136 度），必须立刻熔断暂停，提示“把前腿伸直”。
3. **不俯身熔断 (Trunk Lean)**：用户如果只是站着伸出一条腿，但上身不往下压（躯干倾斜小于 12 度），拉伸感不足。此时 `itemID=39` 会触发拦截，提示“上身向下压”。
