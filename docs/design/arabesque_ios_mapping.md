# arabesque (燕式平衡) 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/arabesque_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 首要考虑复用 iOS 已有的检测项，若无匹配项再分配新 `itemID`。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对静态动作的特殊性，提供明确的状态机实例化与多条件 AND 计时方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- 静态动作不采用动态动作的 `P1/P2` 状态机，而是采用基于 `Holding/Resting` 的多条件 AND 计时状态机。
- 严格遵守 `[min, max, needs_absolute, debounce]` 4 槽位协议。

## 动作业务描述与覆盖结论

**燕式平衡 (arabesque)** 是一项**典型的矢状面单脚支撑静态动作**。
与弓箭步类似，燕式平衡的核心在于躯干的前倾、单腿的支撑以及后腿的高抬。在算法内部（`MetricsCalculator`），可以通过动态判断自动找到支撑腿和抬高腿。因此，**在 iOS 端无需拆分左腿和右腿检测项**。一套通用的检测参数即可兼容所有朝向和支撑侧。

### 训练产物核心检测项（`metrics`）
1. `trunk_lean` (躯干倾斜角) - **核心姿态指标**（评价上身前倾的幅度，目标约 43°，即半俯身状态）。
2. `hip_flexion` (髋关节屈曲/伸展) - **核心姿态指标**（评价后腿抬高程度，目标约 141°）。
3. `knee_flexion` (膝关节屈曲) - **核心支撑指标**（评价支撑腿或抬起腿的伸直程度，目标接近 180°）。

### iOS 复用/新增判定与影响分析

在这三个检测项中，**我们都可以完美复用之前动作定义过的静态 ID**：

- `trunk_lean` -> **强烈建议复用 `itemID=39` (Trunk Lean Static Hold)**。
  - *原因分析*：我们在平板支撑、哥本哈根和弓箭步中都已经使用了 `itemID=39`。燕式平衡要求躯干保持一个前倾角度（通常在 15° 到 60° 之间），底层计算公式（肩中心到髋中心与重力线的夹角）完全一致，仅需下发燕式平衡专属参数即可。**完全可复用**。
- `hip_flexion` -> **强烈建议复用 `itemID=40` (Hip Flexion Static Hold)**。
  - *原因分析*：同样在前面的动作中定义了用于评估髋关节夹角的静态策略。燕式平衡需要它来评估后抬腿与躯干的夹角，底层逻辑通用。**完全可复用**。
- `knee_flexion` -> **强烈建议复用 `itemID=46` (Knee Flexion Static Hold)**。
  - *原因分析*：在上一份弓箭步文档中，我们已经新增了专门针对膝关节角度的纯静态评估 `itemID=46`。燕式平衡要求膝盖尽量伸直，直接复用该 ID 下发新参数即可。**完全可复用**。

## 逐字段 -> 逐索引映射

### 静态 4 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平左右/朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（统一填 15.0 帧，即 0.5 秒，表示需连续达标 0.5s 才进入 Holding） |

---

### itemID=39（Trunk Lean，躯干倾斜角）- 复用
- 来源字段：
  - `normal_range` = `[13.09, 59.88]`
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容朝向差异)

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `13.09` | normal_range 下界 (必须有一定前倾，不能站直) |
| `[1]` | `59.88` | normal_range 上界 (限制前倾过大导致栽倒) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[13.09, 59.88, 1.0, 15.0]`

---

### itemID=40（Hip Flexion，髋关节屈曲/后腿抬高）- 复用
- 来源字段：
  - `normal_range` = `[130.97, 158.54]`
  - `needs_absolute` = `0.0` (夹角天然无向)

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `130.97` | normal_range 下界 (后腿必须抬起，夹角变小) |
| `[1]` | `158.54` | normal_range 上界 (限制站立姿态的180度) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[130.97, 158.54, 0.0, 15.0]`

---

### itemID=46（Knee Flexion，膝关节伸直度）- 复用
- 来源字段：
  - `normal_range` = `[157.37, 192.47]` (转换为合理范围 `[157.37, 180.0]`)
  - `needs_absolute` = `0.0` (夹角天然无向)

由于膝关节的物理极限在180度左右，模型检测的 192.47 是由于轻微过伸或检测误差导致，建议将上限截断至 180.0（或保留 195.0 以容忍误差）。这里保留宽容度。

| iOS item46 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `157.37` | normal_range 下界 (膝盖不能过度弯曲) |
| `[1]` | `195.00` | normal_range 上界 (允许轻微过伸/误差) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[157.37, 195.00, 0.0, 15.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（躯干前倾、后腿高抬、膝盖伸直）
```json
{
  "detectItemIDs": ["39", "40", "46"],
  "detectItemParameters": [
    [13.09, 59.88, 1.0, 15.0],
    [130.97, 158.54, 0.0, 15.0],
    [157.37, 195.00, 0.0, 15.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：`itemID=39`, `itemID=40`, `itemID=46`。所有检测项在之前的静态动作中均已实现。**零新增开发**。
- **需新增开发项**：无。

## 验收点与防作弊/熔断逻辑
1. **未俯身熔断 (Trunk Lean)**：用户直立站着时绝对不能开始计时。必须主动俯身，当躯干倾斜角度进入 13° ~ 60° 区间时，才视为躯干达标。如果前倾过度导致脸朝地，也会触发上限拦截。
2. **腿未抬高熔断 (Hip Flexion)**：俯身后如果不把后腿高高抬起，`hip_flexion` 会接近 180° 或 90°（取决于腿是放下还是前屈），只有当后腿向后上方抬起，使得躯干与大腿夹角在 130° ~ 158° 之间时才达标。
3. **屈膝代偿熔断 (Knee Flexion)**：为了保持平衡，用户可能会严重弯曲支撑腿。一旦膝盖弯曲角度小于 157.37 度，立即熔断暂停。
4. **无视朝向与支撑腿验证**：测试人员应分别尝试“左腿支撑”和“右腿支撑”，并分别尝试面朝屏幕左侧和右侧。系统必须在不需要任何手动切换的前提下，稳定计入时长。
5. **抗抖动（Debounce）验证**：动作过程中难免会晃动，只有完全稳定 0.5 秒（15帧）后，计时器才开始累加。一旦失去平衡导致上述任何一个指标超标，立刻停止计时。
