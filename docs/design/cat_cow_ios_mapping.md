# cat_cow 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/cat_cow_trained.json` 与评估报告 `data/evaluations/cat_cow/20260420_v1.0.json`，给出可直接执行的 iOS 迁移方案。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增 `itemID(21+)`。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- `itemID -> keypoint_profile(17/33)` 由 iOS 注册表维护，不在后端配置中下发。
- 同逻辑但关键点体系不同（17 vs 33）时，必须新建 `itemID`，不能复用旧 `itemID`。
- 本次训练产出的所有核心动作评估和计数阶段，均需支持基于 MP33 骨骼点的 8 槽位动态参数协议。

## 检测项覆盖结论

### 训练产物核心检测项（`metrics`）
- `pelvic_tilt` (骨盆倾斜角) - **主控计数指标** (原训练产物误选为 `knee_flexion`，后经本地测试调优，改用运动幅度更大、表现完美的 `pelvic_tilt` 绝对值作为主控指标)
- `knee_flexion` (膝盖屈角) - **姿态评估指标** (在 `execution` 即塌腰阶段进行评价，确保膝盖稳定贴地)

### iOS 复用/新增判定与影响分析
- `pelvic_tilt` -> **建议新增 `itemID=32`**。
  - *原因分析*：这是主控计数的 metric，不仅需要控制 p1/p2 的 4 个动态流转阈值，还要考虑到不同朝向（左侧身 vs 右侧身）会导致 BlazePose 提取出的骨盆倾角发生正负号反转。**必须在 iOS 侧新增一个专门针对 `pelvic_tilt` 且内置了 `abs()` 绝对值处理逻辑的 8 槽位检测项**，否则一套 JSON 参数无法兼容用户的不同站位。
- `knee_flexion` -> **复用 `itemID=26`（若已有）或新增 `itemID=33`**。
  - *原因分析*：我们在保加利亚蹲中可能已经实现过 `knee_flexion` 的 8 槽位版本（如 `itemID=26`）。如果 `itemID=26` 的逻辑是纯粹的 8 槽位阈值映射且兼容当前需求，可直接复用；如果尚未支持或者内部耦合了其他动作特性，则需要新增 `itemID=33` (`KneeFlexionMP33Strategy`) 用于猫式伸展的执行阶段评价。这里为求严谨与独立，假定新增为 `33`。

### 执行结论（硬规则）
为了保证计数的绝对鲁棒性，**务必在 iOS 侧实现带 `abs()` 处理的 `PelvicTilt` 策略**。核心指标均采用 MP33 动态检测项接入 iOS。

## 新增检测项编号规则
承接之前新动作的分配记录，本动作建议使用以下 ID：
- `32`: `PelvicTiltAbsMP33Strategy`（新增，主计数锚点。**核心：内部获取原始角度后必须先求绝对值！**）
- `33`: `KneeFlexionMP33Strategy`（新增/复用，膝盖稳定度姿态约束）

## 逐字段 -> 逐索引映射

### itemID=32（PelvicTiltAbsMP33，主计数锚点）
- 支持 8 槽位协议（`[0..3]` 相位阈值，`[4..7]` 评估阈值）。
- 极性：`valley_to_peak_to_valley`（绝对值从小变大再变小。弓背时绝对值小，塌腰时绝对值大）。
- **【极其重要】**：iOS 策略在拿到关键点计算出的角度后，**必须先执行 `abs(angle)`**，然后再用得到的值去和下面的阈值做比较与状态机流转。
- 来源字段（采用本地验证通过的手动绝对值参数）：
  - `enter_p1 = 20.0` (绝对值开始变大，超过20°开始塌腰)
  - `exit_p1 = 26.0` (绝对值变大超过26°，塌腰到位)
  - `enter_p2 = 24.0` (绝对值开始减小，跌破24°开始往回弓背)
  - `exit_p2 = 18.0` (绝对值减小跌破18°，弓背完成，计数+1)
  - 评估区间本项暂不参与扣分（可填 0）。

| iOS item32 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `20.0` | enter_p1 (Idle->Rise 阈值) |
| `[1]` | `26.0` | exit_p1 (Rise->Peak 阈值) |
| `[2]` | `24.0` | enter_p2 (Peak->Return 阈值) |
| `[3]` | `18.0` | exit_p2 (Return->Idle 阈值) |
| `[4..7]` | `0.0` | (不参与评估，填0) |

**建议参数行：**
- `[20.0, 26.0, 24.0, 18.0, 0.0, 0.0, 0.0, 0.0]`

---

### itemID=33（KneeFlexionMP33，辅助姿态约束）
- 该项不参与计数控制，仅做姿态评估（在塌腰阶段，评价膝盖是否保持稳定屈曲撑地）。
- 来源字段（自 `metrics[knee_flexion]`）：
  - `normal_range = [81.72, 89.56]`
  - `excellent_range = [83.68, 87.60]`

| iOS item33 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0..3]` | 评估阈值复制 | (如果不参与状态机，通常复制后续4个参数) |
| `[4]` | `metrics...normal_range[0]` | 正常区间下界 |
| `[5]` | `metrics...excellent_range[0]` | 优秀区间下界 |
| `[6]` | `metrics...excellent_range[1]` | 优秀区间上界 |
| `[7]` | `metrics...normal_range[1]` | 正常区间上界 |

**建议参数行：**
- `[81.72, 83.68, 87.60, 89.56, 81.72, 83.68, 87.60, 89.56]`

## 外部配置示例（可人工配置）

### 最小可用（可直接上线试跑，仅包含主控计数）
```json
{
  "detectItemIDs": ["32"],
  "detectItemParameters": [[
    [20.0, 26.0, 24.0, 18.0, 0.0, 0.0, 0.0, 0.0]
  ]]
}
```

### 完整覆盖（含计数主控与膝盖姿态约束）
```json
{
  "detectItemIDs": ["32", "33"],
  "detectItemParameters": [[
    [20.0, 26.0, 24.0, 18.0, 0.0, 0.0, 0.0, 0.0],
    [81.72, 83.68, 87.60, 89.56, 81.72, 83.68, 87.60, 89.56]
  ]]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无
- **需新增开发项**：
  - `itemID=32`（**重点**：Pelvic Tilt 主计数锚点，内部必须增加 `abs()` 取绝对值逻辑）。
  - `itemID=33`（Knee Flexion 姿态约束，视情况可复用）。
- **需补学习项**：
  - 目前的评估报告显示 `error_types` 为空，准确率 (rep_count_acc_at_0) 达到 1.0。若后续业务需要输出具体的动作错误指导语（例如“塌腰幅度不够”等），需要在收集到真实用户错误动作样本后，重新标注训练才能自动生成 ErrorConditions。

## 冲突清单与处理建议
- **冲突1：左右侧视角选侧导致角度正负反转**
  - **冲突描述**：当测试者向左侧身和向右侧身时，BlazePose 算出的骨盆倾斜角会在 `[10°, 40°]` 和 `[-40°, -10°]` 之间翻转，导致状态机彻底死锁卡在 `P1_RISE`。
  - **处理建议**：这是我们在本地 Python 验证阶段遇到的最大陷阱。**在将这个动作迁移给 iOS 开发时，请务必叮嘱他们：在提取 `pelvic_tilt` 用于计数时，一定要套一层 `abs()` 绝对值函数**，然后直接使用这份 JSON 里给出的正数阈值（`[20, 26, 24, 18]`）即可。

## 实施顺序
1. **iOS 开发**：在 iOS 端实现 `itemID=32`，核心是在提取出原始的 `pelvic_tilt` 角度后，立马做一次 `abs()` 处理，再丢进 EMA 滤波与 8 槽位状态机中。
2. **最小可用验证**：通过 Mock JSON 下发 `["32"]`，使用测试集的 `猫式伸展.mp4` 进行正反方向的计数准确率回归，确保不管人朝左还是朝右都能记上数。
3. **完整覆盖与灰度**：接入 `["32", "33"]`，验证姿态约束是否正确扣分。

## 验收点
- **参数对齐**：`detectItemIDs.count == detectItemParameters[0].count`。
- **朝向无关性测试**：找两个猫式伸展视频，一个人物头朝左，一个头朝右，必须都能准确计数（这是对 `abs()` 逻辑的唯一验收标准）。
- **计数准确率**：测试集报告显示当前训练的计数准确率 100%。iOS 上线后应对齐这个准确率，确保放宽后的绝对值阈值能稳健流转状态机。
