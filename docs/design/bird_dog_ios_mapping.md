# bird_dog 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/bird_dog_trained.json` 与本地验证脚本，给出可直接执行的 iOS 迁移方案。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增 `itemID(21+)`。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对鸟狗式作为**左右交替动作**的特殊性，提供明确的状态机实例化与双侧追踪方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- `itemID -> keypoint_profile(17/33)` 由 iOS 注册表维护，不在后端配置中下发。
- 同逻辑但关键点体系不同（17 vs 33）时，必须新建 `itemID`，不能复用旧 `itemID`。
- 本次训练产出的所有核心动作评估和计数阶段，均需支持基于 MP33 骨骼点的 8 槽位动态参数协议。

## 检测项覆盖结论

### 训练产物核心检测项（`metrics`）
- `knee_flexion` (膝关节屈曲) - **主控计数指标**（原训练产物误选为 `trunk_lean`，经本地脚本验证后，已替换为拥有130°完美周期振幅的 `knee_flexion`）
- `hip_flexion` (髋关节屈曲) - **姿态评估指标**（在 Hold 保持阶段评价腿部后伸幅度）
- `trunk_lean` (躯干前倾角) - **姿态评估指标**（在 Hold 保持阶段评价背部是否平直）

### iOS 复用/新增判定与影响分析

由于鸟狗式是**左右交替进行**的动作（例如左手右腿、右手左腿），而在当前的 iOS 8槽位纯数字参数协议中，无法通过字符串传递“检测左侧还是右侧”。为了完美覆盖左右交替的计次需求，**必须为双侧分别分配独立的 itemID**，并在 iOS 端同时实例化两个状态机，最后将两者的 Reps 相加。

- `knee_flexion` -> **建议新增 `itemID=34` (左侧) 和 `itemID=35` (右侧)**。
  - *原因分析*：作为主控计数指标，必须双侧独立追踪。当左腿向后伸展时，触发 `itemID=34` 的状态机流转并计次；当右腿向后伸展时，触发 `itemID=35`。
- `hip_flexion` -> **建议新增 `itemID=36` (左侧) 和 `itemID=37` (右侧)**。
  - *原因分析*：姿态评估同样依赖于当前活动的是哪一侧腿。必须拆分为左右两个检测项。
- `trunk_lean` -> **建议新增 `itemID=38`**。
  - *原因分析*：躯干前倾角是全局姿态指标，计算时依赖肩部和髋部的中点，不区分左右侧，因此只需一个新增的 MP33 检测项。

### 执行结论（硬规则）
为了保证交替动作计数的绝对鲁棒性，**务必在 iOS 侧同时挂载左右两侧的计数检测项，并将两者的 Reps 累加作为最终的动作总次数**。核心指标均采用 MP33 动态检测项接入 iOS。

## 新增检测项编号规则
承接之前新动作的分配记录，本动作建议使用以下 ID：
- `34`: `LeftKneeFlexionMP33Strategy`（新增，左侧主计数锚点）
- `35`: `RightKneeFlexionMP33Strategy`（新增，右侧主计数锚点）
- `36`: `LeftHipFlexionMP33Strategy`（新增，左腿后伸姿态评估）
- `37`: `RightHipFlexionMP33Strategy`（新增，右腿后伸姿态评估）
- `38`: `TrunkLeanMP33Strategy`（新增，全局背部平直度评估）

## 逐字段 -> 逐索引映射

### itemID=34 / 35（Left/Right KneeFlexionMP33，主计数锚点）
- 支持 8 槽位协议（`[0..3]` 相位阈值，`[4..7]` 评估阈值）。
- 极性：`valley_to_peak_to_valley`（从小变大再变小。准备时膝盖着地弯曲角度小，向后伸腿时角度变大接近180°）。
- 来源字段（基于验证通过的修正参数）：
  - `enter_p1 = 120.0` (开始伸腿)
  - `exit_p1 = 150.0` (伸腿到位)
  - `enter_p2 = 140.0` (开始收腿)
  - `exit_p2 = 100.0` (收腿完成，计数+1)
  - 评估区间本项暂不参与扣分（可填 0）。

| iOS item34/35 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `120.0` | enter_p1 (Idle->Rise 阈值) |
| `[1]` | `150.0` | exit_p1 (Rise->Peak 阈值) |
| `[2]` | `140.0` | enter_p2 (Peak->Return 阈值) |
| `[3]` | `100.0` | exit_p2 (Return->Idle 阈值) |
| `[4..7]` | `0.0` | (不参与评估，填0) |

**建议参数行：**
- `[120.0, 150.0, 140.0, 100.0, 0.0, 0.0, 0.0, 0.0]`

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无
- **需新增开发项**：
  - `itemID=34, 35`（双侧 Knee Flexion 主计数锚点，**必须汇总两者 reps**）。
  - `itemID=36, 37`（双侧 Hip Flexion 姿态约束）。
  - `itemID=38`（Trunk Lean 姿态约束）。

## 冲突清单与处理建议
- **冲突1：左右交替计数的底层逻辑冲突**
  - **冲突描述**：iOS 现有的 P1/P2 状态机通常只实例化一个并追踪单个 itemID 决定总 Reps。而鸟狗式要求左右交替，单侧状态机只会漏计一半的动作。
  - **处理建议**：iOS 端需要在 Action 级别（或检测项聚合层）支持**多实例并行计次**。当配置下发了 `["34", "35"]` 时，引擎应分别维护左膝和右膝的独立状态机，并将它们的输出 `rep_count` 进行相加 `total_reps = left_reps + right_reps` 作为该动作的最终计次展示给用户。

## 实施顺序
1. **iOS 开发**：在 iOS 端实现 34~38 号检测项，重点是在动作聚合层支持将 34 和 35 的计数结果相加。
2. **最小可用验证**：通过 Mock JSON 下发 `["34", "35"]`，使用测试集的验证视频进行测试，确认左侧和右侧的伸腿动作均能使右上角的总 Reps 增加。
3. **完整覆盖与灰度**：接入 `["34", "35", "36", "37", "38"]`，验证姿态约束是否正确扣分。

## 验收点
- **参数对齐**：`detectItemIDs.count == detectItemParameters[0].count`。
- **双侧交替计数验收**：播放包含 6 次标准鸟狗式（左3次，右3次）的测试集视频，最终计次必须为 6。
- **姿态评估独立性**：当左腿后伸时，系统应只采用 `itemID=36` (Left Hip Flexion) 进行评价，忽略右侧未动作的腿；反之亦然。这需要在 iOS 侧结合状态机的当前 active 状态进行判断。
