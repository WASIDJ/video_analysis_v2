# copenhagen_plank 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/copenhagen_plank_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对静态动作（Static Action）的特殊性，提供明确的状态机实例化与多条件 AND 计时方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- 静态动作不采用动态动作的 `P1/P2` 状态机，而是采用基于 `Holding/Resting` 的多条件 AND 计时状态机。
- 严格遵守 `[min, max, needs_absolute, debounce]` 4 槽位协议。

## 动作业务描述与覆盖结论

**哥本哈根支撑 (copenhagen_plank)** 是一项**双侧对称的全局静态动作**。与平板支撑类似，它要求身体在矢状面和冠状面都保持笔直。通过算法内部对左右侧的融合/取绝对值处理，我们实现了**动作的全局统一评估，无需区分左侧支撑还是右侧支撑**。即：无论用户面向左还是面向右，都使用同一套检测项配置。

### 训练产物核心检测项（`metrics`）
1. `trunk_lean` (躯干倾斜角) - **核心姿态指标**（评价身体躯干是否保持笔直，避免过度前倾或后仰）。
2. `hip_flexion` (髋关节屈曲) - **核心姿态指标**（评价髋部是否充分伸展，防止“撅屁股”）。
3. `pelvic_tilt` (骨盆倾斜角) - **核心支撑指标**（评价核心和内收肌群是否发力，防止髋部向下塌陷）。

### iOS 复用/新增判定与影响分析

这三个检测项都**不需要**拆分为左/右两个 ID。我们为它们分配全局唯一的静态检测 ID：

- `trunk_lean` -> **复用 `itemID=39` (Trunk Lean Static Hold)**。
  - *原因分析*：我们在平板支撑（plank）中已经定义了 `itemID=39` 用于评估躯干倾斜度。由于哥本哈根支撑同样需要评估躯干的笔直程度，可以直接复用此 ID，仅通过后端下发不同的参数即可。
- `hip_flexion` -> **复用 `itemID=40` (Hip Flexion Static Hold)**。
  - *原因分析*：同样在平板支撑中定义了 `itemID=40` 用于评估髋部是否塌陷或撅起，内置了左右 fallback 逻辑。哥本哈根支撑完全可以复用此逻辑。
- `pelvic_tilt` -> **建议分配 `itemID=45` (Pelvic Tilt Static Hold)**。
  - *原因分析*：虽然动态动作中可能有骨盆倾斜的 ID，但在静态 4 槽位协议下，我们需要一个新的静态骨盆倾斜评估器。通过配置 `needs_absolute=1.0` 抹平向左塌陷或向右塌陷的符号差异，只需一个 ID。

## 逐字段 -> 逐索引映射

### 静态 4 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平左右/朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（统一填 15.0 帧，即 0.5 秒，表示需连续达标 0.5s 才进入 Holding） |

---

### itemID=39（Trunk Lean，躯干倾斜角）
- 来源字段：
  - `normal_range` = `[74.35, 97.94]` (目标值约 85°，接近垂直)
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容朝向)

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `74.35` | normal_range 下界 |
| `[1]` | `97.94` | normal_range 上界 |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[74.35, 97.94, 1.0, 15.0]`

---

### itemID=40（Hip Flexion，髋关节屈曲）
- 来源字段：
  - `normal_range` = `[140.48, 175.03]` (目标值应接近 180°，但允许一定自然弯曲)
  - `needs_absolute` = `0.0` (夹角天然无向，无需绝对值)

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `140.48` | normal_range 下界 |
| `[1]` | `175.03` | normal_range 上界 (限制撅屁股) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[140.48, 175.03, 0.0, 15.0]`

---

### itemID=45（Pelvic Tilt，骨盆倾斜角/塌陷）
- 来源字段：
  - `normal_range` = `[-13.46, 11.62]` (为统一逻辑，后续建议配置直接转为绝对值范围 `[0.0, 15.0]`)
  - `needs_absolute` = `1.0` (必须开启绝对值，兼容左右侧塌陷)

由于我们开启了绝对值，我们需要将原始的 `[-13.46, 11.62]` 转换为绝对值范围，建议取最大偏差，即 `[0.0, 15.0]` 作为最终参数。

| iOS item45 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `15.0` | normal_range 上界 (倾斜容忍度15度) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |

**建议参数行：**
- `[0.0, 15.0, 1.0, 15.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（含躯干、髋关节与骨盆塌陷约束）
```json
{
  "detectItemIDs": ["39", "40", "45"],
  "detectItemParameters": [
    [74.35, 97.94, 1.0, 15.0],
    [140.48, 175.03, 0.0, 15.0],
    [0.0, 15.0, 1.0, 15.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：`itemID=39`, `itemID=40` (假设在平板支撑中已实现)。
- **需新增开发项**：
  - `itemID=45`（Pelvic Tilt 静态检测项：计算骨盆（左右髋连线）与水平面的夹角，需读取 `[2]` 槽位决定是否取绝对值）。
  - **Action 总控层**：确保这 3 个检测项执行强 AND 逻辑，同时达标才走动计时器。

## 验收点与防作弊/熔断逻辑
1. **髋部塌陷熔断 (Pelvic Tilt)**：哥本哈根支撑最容易出现的错误是内收肌无力导致髋部掉落。当骨盆倾斜角度绝对值超过 15 度时，`itemID=45` 会触发拦截，计时器必须立即暂停。
2. **撅屁股熔断 (Hip Flexion)**：用户可能会通过大幅度弯曲髋关节（撅屁股）来借力。当 `hip_flexion` 小于 140.48 度时，`itemID=40` 会触发拦截。
3. **躯干前倾/后仰熔断 (Trunk Lean)**：身体没有保持在一个平面上。当 `trunk_lean` 不在 `[74.35, 97.94]` 范围内时，`itemID=39` 会触发拦截。
4. **无视朝向验证**：要求测试人员分别面向左侧和面向右侧进行测试。由于 `39/45` 的绝对值逻辑和 `40` 的无向角度逻辑，无论面向哪边，都必须能稳定计入时长。
5. **抗抖动（Debounce）验证**：在上下发力的过程中，只有当身体完全稳定在空中超过 0.5 秒（15帧）后，计时器才开始累加。一旦身体掉落，立刻停止计时。
