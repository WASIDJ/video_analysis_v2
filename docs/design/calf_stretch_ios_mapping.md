# calf_stretch (小腿拉伸) 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/calf_stretch_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 延续我们在“腘绳肌拉伸”和“髂胫束拉伸”中使用的 `side_selection_mode = 2.0`（找最直的腿）启发式策略。
- 给出可用于后端人工配置的 JSON 示例。

## 动作业务描述与覆盖结论

**小腿拉伸 (calf_stretch)** 是一项**矢状面（侧面）单侧发力的静态拉伸动作**。
它要求单腿在前方弯曲支撑，另一条腿在后方尽量伸直，以拉伸后腿的小腿肚（腓肠肌）。同时躯干可能保持直立或随势前倾。
由于这又是一个极度不对称的单侧动作，我们在 iOS 端无需拆分左腿和右腿检测项。一套通用的检测参数，结合 `side_selection_mode = 2.0`（找最直的那条腿），即可兼容左右腿及人物朝向的所有情况。

## 关键边界与协议说明

### 5 槽位寻边策略 (`[min, max, needs_absolute, debounce, side_selection_mode]`)
与腘绳肌拉伸极其相似，小腿拉伸也是“一条腿弯，一条腿直”。我们要评估的拉伸目标，正是那条**在后面绷直的腿**。
因此，对于所有需要选腿的双侧指标（如 `knee_flexion` 和 `hip_flexion`），我们直接下发 **`side_selection_mode = 2.0`**（静态启发式 - 选角度最大的一侧）。
iOS 端的底层状态机会在寻边期自动越过弯曲的支撑腿，死死锁住后方那条相对较直的拉伸腿。

### iOS 复用判定与分析（100% 复用！）

在这套设计下，所有的指标都完美复用了我们在前几个动作中规划的静态检测项。

- `knee_flexion` (膝关节屈曲) -> **复用 `itemID=46` (Knee Flexion Static Hold)**。
  - 核心指标。要求后侧腿伸直。双侧指标，第 5 槽位下发 **`2.0`**（找更直的腿）。
- `trunk_lean` (躯干前倾角) -> **复用 `itemID=39` (Trunk Lean Static Hold)**。
  - 全局脊柱指标，不涉及选腿，第 5 槽位填 `0.0`。
- `hip_flexion` (髋关节屈曲) -> **复用 `itemID=40` (Hip Flexion Static Hold)**。
  - 双侧指标，跟随 `knee_flexion` 找到的那条直腿计算髋部夹角。第 5 槽位下发 **`2.0`**。

## 逐字段 -> 逐索引映射

### itemID=46（Knee Flexion，膝关节屈曲）- 复用
- 来源字段：
  - `normal_range` = `[111.67, 147.63]` (小腿拉伸时后腿必须尽量伸直，这里将上限放宽至 185.0 容忍检测过伸，下限为了容忍柔韧性差的人卡在 110.0)
  - `needs_absolute` = `0.0`

| iOS item46 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `110.0` | normal_range 下界 (必须达到一定的伸直度) |
| `[1]` | `185.0` | normal_range 上界 (允许后腿完全伸直甚至微小过伸) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `2.0` | **核心：指示策略强制去寻找左/右膝中【角度更大（更直）】的那条腿作为拉伸腿** |

**建议参数行：**
- `[110.0, 185.0, 0.0, 15.0, 2.0]`

---

### itemID=39（Trunk Lean，躯干前倾角）- 复用
- 来源字段：
  - `normal_range` = `[-11.09, 21.05]` (转换为绝对值后为 `[0.0, 25.0]`，允许上身直立或轻微前倾)
  - `needs_absolute` = `1.0`

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `25.0` | normal_range 上界 (允许躯干最多前倾 25度) |
| `[2]` | `1.0` | **开启绝对值处理 (抹平朝左朝右的差异)** |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `0.0` | 全局脊柱指标，填 0.0 |

**建议参数行：**
- `[0.0, 25.0, 1.0, 15.0, 0.0]`

---

### itemID=40（Hip Flexion，髋关节屈曲）- 复用
- 来源字段：
  - `normal_range` = `[105.79, 140.27]` (限制拉伸腿的髋部不要过度折叠)
  - `needs_absolute` = `0.0`

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `105.79` | normal_range 下界 |
| `[1]` | `145.0` | normal_range 上界 |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `2.0` | **跟随选腿逻辑，寻找角度最大的一侧计算髋部** |

**建议参数行：**
- `[105.79, 145.0, 0.0, 15.0, 2.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（含后腿伸直、躯干不严重弯腰、髋部展开）
```json
{
  "detectItemIDs": ["46", "39", "40"],
  "detectItemParameters": [
    [110.0, 185.0, 0.0, 15.0, 2.0],
    [0.0, 25.0, 1.0, 15.0, 0.0],
    [105.79, 145.0, 0.0, 15.0, 2.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无。
- **需新增开发项**：
  - 前置依赖：必须完成在“股四头肌拉伸”和“腘绳肌拉伸”中规划的 `side_selection_mode` 5 槽位寻边状态机。尤其是支持 `mode == 2.0`（找角度最大的一侧）的相对阈值锁定逻辑。只要底层完工，本动作即可零代码接入！

## 验收点与防作弊/熔断逻辑
1. **拉伸腿识别验收（核心）**：测试人员故意做出夸张的前腿弯曲（90度）和后腿伸直（180度）。由于下发了 `side_selection_mode = 2.0`，系统必须立刻锁定那条在后面伸直的腿。
2. **屈膝代偿熔断 (Knee Flexion)**：用户在拉伸小腿时如果后腿膝盖弯曲（角度小于 110 度），拉伸力会错误地转移到比目鱼肌，失去拉伸腓肠肌的效果。此时 `itemID=46` 必须立刻熔断暂停，提示“后腿膝盖绷直”。
3. **过度弯腰熔断 (Trunk Lean)**：用户如果不往前推墙，而是直接向下弯腰（躯干倾斜超过 25 度），不仅拉伸不到小腿，还会给腰椎带来压力。此时 `itemID=39` 会触发拦截，提示“上身保持挺直”。
