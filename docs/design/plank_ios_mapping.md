# plank 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/plank_static_config.json` 与本地验证脚本，给出可直接执行的 iOS 迁移方案。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增 `itemID(39+)`。
- 给出可用于后端人工配置的 `detectItemIDs + detectItemParameters` 示例。
- 针对静态动作（Static Action）的特殊性，提供明确的状态机实例化与多条件 AND 计时方案。

## 关键边界
- iOS 侧不定义动作，不固化动作参数，只消费后端下发的 `detectItemIDs + detectItemParameters`。
- 静态动作不采用动态动作的 `P1/P2` 状态机，而是采用基于 `Holding/Resting` 的多条件 AND 计时状态机。
- 本次训练产出的所有核心动作评估和计数阶段，均需支持基于 MP33 骨骼点的 8 槽位静态参数协议（前 4 个槽位有效）。

## 检测项覆盖结论

### 训练产物核心检测项（`metrics`）
- `trunk_lean` (躯干前倾角) - **核心姿态指标**（评价背部是否平直，目标角度约 85°）
- `hip_flexion` (髋关节屈曲) - **核心姿态指标**（评价臀部是否过高或塌陷，目标角度约 137°）

### iOS 复用/新增判定与影响分析

平板支撑是典型的**双侧对称静态动作**。在现有的 iOS MP33 架构中，我们需要分配专门的静态检测项 ID。

- `trunk_lean` -> **建议新增 `itemID=39` (Trunk Lean Static Hold)**。
  - *原因分析*：躯干前倾角是全局姿态指标，计算时依赖肩部和髋部，不区分左右侧，因此只需一个新增的 MP33 静态检测项。
- `hip_flexion` -> **建议新增 `itemID=40` (Hip Flexion Static Hold)**。
  - *原因分析*：由于人是侧对镜头或正对镜头，我们需要一个能够评价髋关节角度的静态策略。计算时若一侧被遮挡，应当 fallback 到另一侧（在 iOS 端实现，或者分别下发左右两个 ID 并要求任一达标即可）。为了简化逻辑并保证最高容错率，建议实现为一个综合的双侧 fallback 策略（即 `itemID=40` 内部优先计算左侧，左侧置信度低则计算右侧）。

### 执行结论（硬规则）
为了保证计时的绝对鲁棒性，**务必在 iOS 侧同时挂载 39 和 40 两个检测项，并在动作总控层执行强 AND 逻辑（即 39 和 40 必须同时处于 Holding 状态，总计时器才累加）**。核心指标均采用 MP33 动态检测项接入 iOS。

## 新增检测项编号规则
承接之前新动作的分配记录，本动作建议使用以下 ID：
- `39`: `TrunkLeanStaticHoldMP33Strategy`（新增，全局背部平直度评估）
- `40`: `HipFlexionStaticHoldMP33Strategy`（新增，臀部高低/塌陷评估，内置左右 fallback）

## 逐字段 -> 逐索引映射

### 静态 8 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（固定填 0.5，表示需连续达标 0.5s 才进入 Holding） |
| `[4..7]` | 预留 | 填 `0.0` |

### itemID=39（Trunk Lean Static Hold，背部平直度）
- 来源字段（基于验证通过的修正参数）：
  - `normal_range` = `[75.45, 94.12]`
  - `needs_absolute` = `1.0` (需要取绝对值，保证朝左朝右一致)

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `75.45` | normal_range 下界 |
| `[1]` | `94.12` | normal_range 上界 |
| `[2]` | `1.0` | 开启绝对值处理 |
| `[3]` | `0.5` | 0.5秒防抖 |
| `[4..7]` | `0.0` | (不参与评估，填0) |

**建议参数行：**
- `[75.45, 94.12, 1.0, 0.5, 0.0, 0.0, 0.0, 0.0]`

---

### itemID=40（Hip Flexion Static Hold，臀部塌陷/过高约束）
- 来源字段（基于验证通过的修正参数）：
  - `normal_range` = `[120.97, 180.00]`
  - `needs_absolute` = `0.0` (基于无向向量点积 arccos，天然无视朝向，无需额外取绝对值)

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `120.97` | normal_range 下界 |
| `[1]` | `180.00` | normal_range 上界 |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `0.5` | 0.5秒防抖 |
| `[4..7]` | `0.0` | (不参与评估，填0) |

**建议参数行：**
- `[120.97, 180.00, 0.0, 0.5, 0.0, 0.0, 0.0, 0.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（含背部与臀部双重约束）
```json
{
  "detectItemIDs": ["39", "40"],
  "detectItemParameters": [[
    [75.45, 94.12, 1.0, 0.5, 0.0, 0.0, 0.0, 0.0],
    [120.97, 180.00, 0.0, 0.5, 0.0, 0.0, 0.0, 0.0]
  ]]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无
- **需新增开发项**：
  - `itemID=39`（Trunk Lean 静态检测项，需读取 `[2]` 槽位决定是否取绝对值）。
  - `itemID=40`（Hip Flexion 静态检测项，需内置左/右侧 fallback 逻辑：优先读左腿，置信度极低读右腿）。
  - **Action 总控层改造**：必须新增静态动作强 AND 聚合逻辑（当 39 和 40 同时 Holding 时，总计时器累加）。

## 实施顺序
1. **iOS 开发**：在 iOS 端基于新的静态规范文档实现 39、40 号检测项，并完成 Action 聚合层的强 AND 计时逻辑改造。
2. **Mock 验证**：通过 Mock JSON 下发 `["39", "40"]`，使用测试集的验证视频进行测试，确认只有当人物完全进入平板支撑姿态 0.5 秒后，进度条才开始走动；一旦塌腰，进度条立即停止并报错。
3. **朝向兼容验证**：播放向左趴下和向右趴下的视频，确保都能稳定计时。

## 验收点
- **参数对齐**：`detectItemIDs.count == detectItemParameters[0].count`。
- **抗抖动验收**：在动作进入的瞬间，不应该立即跳秒；只有稳定维持在合理区间（75°~94° 之间）超过 0.5 秒后，计时才生效。
- **塌腰/撅屁股熔断**：当测试者故意将臀部抬高（`hip_flexion` < 120°）或塌陷（`hip_flexion` > 180°）时，总计次必须立即暂停，直到姿势恢复。
