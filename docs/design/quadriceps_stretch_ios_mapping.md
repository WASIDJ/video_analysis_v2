# quadriceps_stretch (股四头肌拉伸) 到 iOS 检测项映射规范

## 文档目的
- 基于训练产物 `config/action_configs/quadriceps_stretch_static_config.json`，给出可直接执行的 iOS 迁移方案。
- 引入 `SideSelectionMode` 启发式策略配置，解决纯静态动作的“主导侧（拉伸腿）”识别问题。
- 明确哪些检测项可复用 iOS 现有 `itemID`，哪些必须新增或升级。
- 给出可用于后端人工配置的 JSON 示例。

## 关键边界与协议升级说明

### 原有 4 槽位协议的局限性
在处理深蹲、侧抬腿等动作时，iOS 策略可以根据“关键点可见度”或“Y 轴运动幅度”自动找到发力腿。但**股四头肌拉伸是一个纯静态的单侧动作**，支撑腿和拉伸腿的 Y 轴在测试窗口内**完全没有运动幅度**，这会导致原有的自动选边逻辑失效（通常会默认选中右脚），进而导致判定失败。

### 升级为 5 槽位协议 (`[min, max, needs_absolute, debounce, side_selection_mode]`)
为了让检测项具备在纯静态动作中找对目标腿的能力，我们将原有的 4 槽位下发数组扩展为 **5 槽位**，新增第 5 个参数 `side_selection_mode`（选边模式），定义如下：
- `0.0`: **默认模式**（动态动作常用的逻辑：选置信度高的、或者运动幅度大的）。
- `1.0`: **静态启发式 - 选角度最小的一侧** (Min Angle)。适用于找极度弯曲的关节，如股四头肌拉伸的拉伸腿。
- `2.0`: **静态启发式 - 选角度最大的一侧** (Max Angle)。
- `3.0`: **静态启发式 - 选 Y 轴坐标更高的一侧** (Max Y)。适用于单脚站立找抬起来的脚。

## 动作业务描述与覆盖结论

**股四头肌拉伸 (quadriceps_stretch)** 是一项**矢状面单侧发力的静态拉伸动作**。
它要求单腿站立，另一条腿向后极度屈膝（脚跟贴近臀部），同时躯干直立，大腿不往前屈。由于引入了上述的 `side_selection_mode` 协议，**在 iOS 端无需拆分左腿和右腿检测项**。一套通用的检测参数即可兼容“拉伸左腿”和“拉伸右腿”以及“人物朝左”和“人物朝右”的所有情况。

### 训练产物核心检测项（`metrics`）
1. `knee_flexion` (膝关节屈曲) - **核心拉伸指标**（评价拉伸腿是否充分弯曲）。
2. `trunk_lean` (躯干倾斜角) - **核心防代偿指标**（评价身体是否直立，防止弯腰减轻拉伸感）。
3. `hip_flexion` (髋关节屈曲) - **核心防代偿指标**（评价拉伸侧大腿是否前屈，要求大腿与躯干尽量保持在同一直线上）。

### iOS 复用/新增判定与影响分析

- `trunk_lean` -> **复用 `itemID=39` (Trunk Lean Static Hold)**。
  - *原因分析*：这是计算躯干与重力线夹角的通用策略，和弓箭步的逻辑完全一致。这是一个全局指标（算脊柱），不涉及选腿，因此第 5 槽位填 `0.0`。**完全可复用**。
- `knee_flexion` -> **复用并升级 `itemID=46` (Knee Flexion Static Hold)**。
  - *原因分析*：在弓箭步中定义了静态膝关节角度计算策略。但现在它**必须升级以支持读取第 5 槽位的 `side_selection_mode`**。对于本动作，它的第 5 槽位必须下发 `1.0`（找角度最小的一侧）。
- `hip_flexion` -> **复用并升级 `itemID=40` (Hip Flexion Static Hold)**。
  - *原因分析*：在之前的动作中用于计算肩-髋-膝夹角。由于它也涉及选腿（我们要看的是被拉伸那条腿的大腿前屈情况），因此它也**必须升级以支持读取第 5 槽位**。同样，它跟随 `knee_flexion` 找到的拉伸腿来计算，所以也下发 `1.0`（或者在底层策略中，所有双侧指标共享同一次启发式寻边结果）。

## 逐字段 -> 逐索引映射

### 静态 5 槽位协议定义
| 索引 | 语义定义 | 说明 |
|---|---|---|
| `[0]` | `normal_range.min` | 合格区间下界 |
| `[1]` | `normal_range.max` | 合格区间上界 |
| `[2]` | `needs_absolute` | 是否取绝对值（1.0 是，0.0 否，抹平左右/朝向差异） |
| `[3]` | `debounce_seconds` | 抗抖动阈值（统一填 15.0 帧，即 0.5 秒） |
| `[4]` | `side_selection_mode` | **[新增]** 选边启发式模式（1.0 为选小角度侧，0.0 为默认） |

### itemID=46（Knee Flexion，膝关节屈曲）- 升级
- 来源字段：
  - `normal_range` = `[30.35, 85.66]` (上限 85° 为了容忍柔韧性差的人，但越小越好)
  - `needs_absolute` = `0.0`

| iOS item46 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `30.35` | normal_range 下界 |
| `[1]` | `85.66` | normal_range 上界 |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `1.0` | **核心：指示策略强制去寻找左/右膝中角度更小的那条腿作为拉伸腿** |

**建议参数行：**
- `[30.35, 85.66, 0.0, 15.0, 1.0]`

### itemID=39（Trunk Lean，躯干倾斜角）- 复用
- 来源字段：
  - `normal_range` = `[-4.41, 18.43]` (转换为绝对值后设定为 `[0.0, 18.5]`)
  - `needs_absolute` = `1.0`

| iOS item39 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `0.0` | normal_range 下界 |
| `[1]` | `18.5` | normal_range 上界 (允许躯干最多倾斜 18.5度) |
| `[2]` | `1.0` | **开启绝对值处理** |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `0.0` | 全局脊柱指标，不涉及选腿，填 0.0 |

**建议参数行：**
- `[0.0, 18.5, 1.0, 15.0, 0.0]`

### itemID=40（Hip Flexion，髋关节屈曲）- 升级
- 来源字段：
  - `normal_range` = `[144.11, 191.67]` (截断合理上限至 185.0)
  - `needs_absolute` = `0.0`

| iOS item40 索引 | 来源字段 | 说明 |
|---|---|---|
| `[0]` | `144.11` | normal_range 下界 (大腿允许有一定前屈) |
| `[1]` | `185.0` | normal_range 上界 (限制过伸) |
| `[2]` | `0.0` | 关闭绝对值处理 |
| `[3]` | `15.0` | 0.5秒防抖 |
| `[4]` | `1.0` | **跟随找拉伸腿的逻辑，强制去找膝盖角度小的那一侧来计算髋部夹角** |

**建议参数行：**
- `[144.11, 185.0, 0.0, 15.0, 1.0]`

## 外部配置示例（可人工配置）

### 完整覆盖（含拉伸深度、躯干直立、大腿不前屈）
```json
{
  "detectItemIDs": ["46", "39", "40"],
  "detectItemParameters": [
    [30.35, 85.66, 0.0, 15.0, 1.0],
    [0.0, 18.5, 1.0, 15.0, 0.0],
    [144.11, 185.0, 0.0, 15.0, 1.0]
  ]
}
```

## 已可直接上线项 vs 需新增开发项
- **已可直接上线项**：无。
- **需新增开发项（底层协议升级与状态机改造）**：
  
  **基础策略类改造（寻边与锁定状态机）**：
  所有的双侧静态策略（如 `itemID=46` 和 `itemID=40`）必须扩展以支持读取 `params[4]` 的 `side_selection_mode`，并引入**基于相对差值的“选边锁定与重置”（Lock & Reset）机制**。
  
  - **1. 寻边期 (Unlocked)**：
    当 `lockedSide == nil`，且读取到 `mode == 1.0` 时，策略在每帧计算左腿和右腿的夹角。
    **触发特征（相对差值）**：不使用绝对阈值（如小于90度），而是判断 `abs(leftAngle - rightAngle) > 20.0`。如果两腿角度差大于 20 度，说明用户已经做出了明显的单腿屈膝动作（排除了站立时的轻微抖动和误差）。
  - **2. 锁定与死锁追踪 (Locked)**：
    一旦触发上述相对差值条件，立刻将 `lockedSide` 赋值为角度较小的那一条腿（`leftAngle < rightAngle ? .left : .right`）。在后续的帧中，**不再重新寻边**，只死锁计算并评估这条被锁定的腿。这防止了中途换脚继续累加时长的作弊漏洞。
  - **3. 动作中断与重置 (Reset)**：
    当用户放下腿，导致锁定侧的指标超出合格范围（例如回到 180°），计时器会中断并清零 `current_holding_frames`。此时可以将 `lockedSide` 重置为 `nil`，重新进入寻边期。
  
  - **Action 总控层**：确保这 3 个检测项执行强 AND 逻辑。

## 验收点与防作弊/熔断逻辑
1. **拉伸腿识别验收（核心）**：测试人员分别拉伸左腿和右腿。因为下发了 `side_selection_mode = 1.0`，系统必须能够准确识别出弯曲度更大的那条腿为拉伸腿，而不能错误地锁定直立的支撑腿。
2. **弯腰代偿熔断 (Trunk Lean)**：用户拉伸时，如果为了手够到脚而大幅度弯腰（倾斜超过 18.5 度），必须立刻熔断暂停，提示“保持上身直立”。
3. **大腿前屈代偿熔断 (Hip Flexion)**：用户如果把拉伸腿的大腿严重往前抬（小于 144度），拉伸感会消失。此时 `itemID=40` 会触发拦截，提示“大腿向后伸”。
4. **抗抖动（Debounce）验证**：抓脚过程中的摇晃不计入时间。只有单脚站稳，且三个指标全部在合理范围内保持 0.5 秒（15帧）后，计时器才开始累加。一旦失去平衡，立刻停止计时。
